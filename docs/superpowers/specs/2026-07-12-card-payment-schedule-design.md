## Контекст и цель

Финальное ревью 3a выявило: блок карт в `buildForecast` ставит по карте ОДНО ближайшее
обязательство. Для `payStrategy: 'minimum'` это неверно — минимальные платежи идут каждый
месяц до закрытия долга, проценты капают на остаток. Из-за этого прогноз недооценивает
нагрузку карт, а сравнение переносов (3a) выглядит искажённо (перенос долга численно
кажется невыгодным). Этот под-этап (3b-1) чинит модель в ядре.

3b разбит: 3b-1 (этот спек) — график погашения в ядре; 3b-2 (контур) — вкладка "Карты:
стратегия" (календарь, общая цена, свободный лимит). 3c (контур) — оптимизатор.

## Модель графика погашения (payStrategy: minimum)

Карта с `minimum` разворачивается в РЯД ежемесячных платежей по циклам карты, пока долг не
закрыт или не кончился горизонт. Каждый шаг (месяц):
- Проценты за месяц: `interest = apr × остаток × statementCycleDays/365` (начисляются на
  остаток; для minimum грейс не спасает — минимальные платежи это признание долга под
  проценты).
- Платёж: `pay = cardMinCore(card, остаток, rates)` — минимум от ТЕКУЩЕГО остатка.
- Тело: `principalPaid = max(0, pay − interest)`; `остаток −= principalPaid`.
- Событие на дату `due` каждого цикла (через `cardCycle`).
- Стоп: остаток ≤ 0 (последний платёж = остаток + проценты, гасит всё) ИЛИ дата > конца
  горизонта. Хвост непогашенного долга остаётся (в календаре/цене 3b-2 покажем остаток).
- Guard от бесконечного цикла (как в cardCycle).

`payStrategy: 'full'` НЕ меняется: одно событие, весь долг на `due` в грейс без процентов.

## Чистые функции (`src/finance.js`)

Вводим ДВЕ чистые функции, чтобы избежать задвоения процентов в графике:

### cardMinCore(card, balanceRub, rates) → number

Тело минимального платежа БЕЗ процентов (только % от остатка или фикс, кламп до остатка):
```
byPct = balanceRub × minPaymentPercent/100
fixed = moneyToRub(minPaymentFixed, rates)
return min(max(byPct, fixed), balanceRub)
```
Используется в `cardPaymentSchedule` (проценты там начисляются ОТДЕЛЬНО отдельным шагом,
поэтому core берётся без них — иначе у карт с minPaymentPlusInterest=true проценты
задвоятся).

### cardMinPayment(card, rates) → number (рефактор, поведение сохранено)

Становится: `cardMinCore(card, cardDebt(card, rates), rates)` ПЛЮС проценты, если
`minPaymentPlusInterest`:
```
const debt = cardDebt(card, rates)
const core = cardMinCore(card, debt, rates)
const interest = minPaymentPlusInterest ? apr × debt × statementCycleDays/365 : 0
return min(core + interest, debt)
```
Это ровно текущая формула cardMinPayment, просто core вынесен в cardMinCore. Все тесты
этапа 1 (Т-Банк 14%/600, Озон 4%+проценты/400, Уралсиб 3%+проценты/300, кламп до долга)
должны остаться зелёными — проверить регрессией.

### cardPaymentSchedule(card, rates, from, end) → [{ date, amount, remainingAfter, interest }]

Ряд событий погашения карты по стратегии minimum до закрытия долга или конца горизонта.
- Стартовый остаток = `cardDebt(card, rates)`.
- Идёт по циклам `cardCycle`, для каждого цикла: считает interest, платёж (core-минимум),
  тело, уменьшает остаток, добавляет `{ date: due, amount: платёж, remainingAfter, interest }`.
- Стоп по остатку ≤ 0 или дате > end. Guard.
- Возвращает пустой массив, если долг ≤ 0.

## Тесты (`test/finance.test.js`)

- `cardMinCore`: минимум от произвольного остатка (% и фикс, кламп до остатка). cardMinPayment
  как частный случай даёт прежний результат (регрессия).
- `cardPaymentSchedule`: карта с долгом minimum даёт НЕСКОЛЬКО событий; остаток убывает
  монотонно; сумма тел ≤ исходного долга; проценты положительны для карт с apr>0. Долг,
  не закрываемый в горизонте, обрывается на конце горизонта (хвост остаётся). Долг ≤ 0 →
  пустой массив.
- `buildForecast`: карта minimum даёт несколько card-событий в горизонте (было одно);
  карта full — одно событие (регрессия не сломана). Общий прогон 51+ тестов зелёный.
