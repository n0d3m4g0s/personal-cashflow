# Этап 3a: модель переводов между картами + ход переноса долга

Дата: 2026-07-12
Статус: согласован, готов к плану реализации
Предшествует: этап 1 (реальные даты карт) и этап 2 (движок сценариев) — оба в main.

## Контекст и цель

Этап 3 (стратегия перекрытия кредиток) разбит на три под-этапа:
- 3a (этот спек): модель переводов + ход переноса долга в движке сценариев.
- 3b (контур): вкладка "Карты: стратегия" — календарь обязательств по всем картам,
  общая цена обслуживания, отображение.
- 3c (контур): оптимизатор-советник — ищет план погашения с МИНИМАЛЬНЫМ РИСКОМ
  ПРОСРОЧКИ, предлагает переливы с объяснением.

3a — фундамент: честная модель "перенести долг с карты на карту". Ключевой факт (из
ресёрча и данных пользователя): грейс на перевод есть ТОЛЬКО у карт Т-Банк; у Озон/Уралсиб
перевод вне грейса (проценты с первого дня). При этом карта мужа Т-Банк заполнена (свободно
~6k из 238k), поэтому реальный приёмник долга — карта жены (свободно ~150k). "Карусель"
переносов невозможна: перелив лишь выигрывает время до грейса карты жены, долг всё равно
надо гасить деньгами.

## Модель карты: новые поля (в дополнение к этапу 1)

```js
{
  // этап 1 (уже есть)
  transferLimit: { amount, currency },   // беспроцентный лимит перевода
  transferGraceDays: 55,                 // грейс на перевод (дней)

  // этап 3a (новые)
  transferGraceEnabled: true,            // действует ли грейс на перевод ВООБЩЕ
                                         //   Т-Банк: true; Озон/Уралсиб: false
  transferFeePercent: 0,                 // комиссия за перевод, % (Т-Банк сверх лимита 2.9)
  transferFeeFixed: { amount, currency },// фикс. комиссия (Т-Банк 290 руб)
}
```

`migrate` для старых карт: `transferGraceEnabled: false` (консервативно), `transferFeePercent: 0`,
`transferFeeFixed: {amount:0,currency:'RUB'}`. Идемпотентно (== null гварды).

Реальные значения для карт пользователя (правятся отдельно, вне 3a-кода):
- Т-Банк жена: transferGraceEnabled true, transferLimit 150000, transferFeePercent 2.9,
  transferFeeFixed 290.
- Т-Банк муж: transferGraceEnabled true, но свободный лимит ~6k (долг почти под завязку).
- Озон, Уралсиб: transferGraceEnabled false (грейс на перевод не действует).

## Расчёт стоимости переноса (чистая функция)

```js
// toCard — карта, НА которую переезжает долг (она платит цену перевода).
transferCost(toCard, amount, transferDate, repayDate, rates)
  → { fee, interest, total, availableLimit, exceedsLimit }
```

- `availableLimit` = min(transferLimit, creditLimit − currentDebt) карты toCard (в рублях).
  Физически доступная сумма перевода на эту карту.
- `exceedsLimit` = amount > availableLimit (флаг предупреждения для UI).
- `fee` = transferFeePercent/100 × amount + transferFeeFixed (в рублях).
- `interest`:
  - если transferGraceEnabled === false → проценты на ВСЮ сумму с первого дня:
    `apr × amount × дни(transferDate→repayDate)/365`.
  - если transferGraceEnabled === true → та же логика, что cardLoanInterest этапа 2:
    free = min(amount, transferLimit) беспроцентно при возврате в грейс перевода
    (transferDate + transferGraceDays); over = превышение под проценты с первого дня;
    если возврат позже грейса — проценты на free за дни сверх.
- `total` = fee + interest.

Обобщение cardLoanInterest: текущая cardLoanInterest (этап 2) неявно предполагает
transferGraceEnabled=true. Обобщаем её так, чтобы при transferGraceEnabled=false вся сумма
шла под проценты. У карты жены (единственный cardLoan этапа 2) transferGraceEnabled=true,
поэтому поведение cardLoan НЕ меняется, тесты этапа 2 остаются зелёными. transferCost
переиспользует эту обобщённую процентную логику, добавляя комиссию и лимит.

## Ход переноса долга (движок сценариев)

```js
// Перенести долг amount С карты fromCardId (её долг снимаем/гасим)
// НА карту toCardId (её долг растёт, она платит за перевод) в дату date.
{ type: 'transfer', fromCardId, toCardId, amount: {amount, currency}, date,
  repay: 'auto' | 'manual', repayDate }
```

Терминология (ВАЖНО, совпадает с бытовым "перенести долг с Озона на карту жены"):
- `fromCardId` — карта, С КОТОРОЙ долг снимаем (её долг ГАСИТСЯ этим переносом). Пример:
  Озон. В прогнозе уменьшается её обязательство.
- `toCardId` — карта, НА КОТОРУЮ долг переезжает (её долг РАСТЁТ, она платит цену перевода:
  комиссию + проценты по своим правилам). Пример: карта жены. Именно у ЭТОЙ карты нужен
  свободный лимит и именно её грейс/ставка определяют цену.
- Расчёт цены: `transferCost(toCard, amount, ...)` — источник денег и стоимости это карта
  `toCardId` (куда переезжает долг), НЕ fromCardId.

applyScenario для transfer (модель A, деньги не идут в наличные):
- fromCardId: currentDebt уменьшается на amount (долг снят/погашен переносом). В прогнозе
  это снижает обязательство этой карты.
- toCardId: currentDebt НЕ трогаем в applyScenario (модель A этапа 2 — иначе двойной счёт).
  Рост долга toCardId и его возврат моделируются в evaluateScenario парным событием.
- Наличные (+amount income) НЕ добавляются: деньги ушли прямо на гашение fromCardId, на руки
  не попадают. Нетто по наличным от самого перевода = 0; влияние на прогноз идёт через
  изменение обязательств карт (fromCardId платит меньше, toCardId — больше).

evaluateScenario для transfer:
- Уменьшить обязательство fromCardId (уменьшить его currentDebt в форке до buildForecast —
  эта карта погашена переносом, платит меньше/ничего).
- toCardId: определить дату возврата долга ДЕНЬГАМИ (авто по порогу balance≥amount+buffer
  или ручная), поставить событие-расход −amount на дату возврата (гасим новый долг toCardId).
- Цена: transferCost(toCard, ...) добавляется в overpayment сценария.
- graceOk для transfer: возврат ≤ грейса перевода toCardId (если у toCardId grace enabled).

Учёт свободного лимита: если amount > availableLimit карты toCardId (min(transferLimit,
свободный лимит toCardId)), перенос всё равно считается, но в метриках/UI ставится флаг
exceedsLimit (физически невозможный объём — предупреждение).

## Тесты (`test/scenarios.test.js`)

- transferCost(toCard=карта жены): в пределах лимита, возврат в грейс → total = fee
  (interest 0). Сверх лимита → проценты на превышение. toCard без грейса на перевод
  (transferGraceEnabled=false, напр. Озон как приёмник) → проценты на всю сумму с первого
  дня. Комиссия (% + фикс) считается верно.
- availableLimit/exceedsLimit: перенос больше min(transferLimit, свободный лимит toCard) →
  флаг exceedsLimit true.
- applyScenario transfer (fromCardId=Озон, toCardId=жена): долг fromCardId (Озон)
  уменьшается на amount; долг toCardId (жена) НЕ растёт в applyScenario (модель A); наличные
  не добавляются.
- evaluateScenario transfer: перенос долга Озона (from) на карту жены (to) с возвратом в
  грейс → overpayment = комиссия (проценты 0); перенос на карту без грейса (to без grace) →
  overpayment с процентами. Метрики не падают.
- Обобщённая cardLoanInterest: тест, что при transferGraceEnabled=false вся сумма под
  проценты; при true (или отсутствии поля для обратной совместимости cardLoan) — прежнее
  поведение. Существующие тесты cardLoan этапа 2 остаются зелёными.
- Регрессия нетто: перенос сам по себе (без цены) не создаёт лишнего движения наличных.

## Границы 3a (YAGNI)

НЕ входит: вкладка-советник "Карты: стратегия" (3b), оптимизатор (3c), лимиты день/месяц,
цепочки переносов как отдельная фича (пользователь может добавить несколько ходов transfer
вручную — движок посчитает каждый честно), правка реальных данных пользователя. Только:
модель полей + transferCost + ход transfer + редактор хода в существующей вкладке "Сценарии"
+ тесты.
