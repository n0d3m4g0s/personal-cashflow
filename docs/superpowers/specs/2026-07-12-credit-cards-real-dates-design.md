# Этап 1: кредитки как единый организм — реальные даты грейса и авторасчёт платежей

Дата: 2026-07-12
Статус: согласован, готов к плану реализации

## Контекст и проблема

Семейный финансовый менеджер (Vue 3 + Vite, статика, localStorage) считает денежный
поток по дням. Модель кредитных карт недоработана:

1. Даты цикла задаются абстрактным "числом месяца" (`statementDay`/`dueDay`). Это
   непрозрачно и ломается на переходах через месяц: нельзя задать реальное
   "выписка 26 июля, платёж до 19 августа", когда грейс переваливает через границу
   месяца.
2. Обязательный платёж считается по одной грубой формуле (% от долга), тогда как у
   банков формулы разные (Озон 4% + проценты, не менее 400; Уралсиб 3% + проценты, не
   менее 300; Т-Банк % от долга).
3. Пользователь сообщает, что "кредитки не учитываются в прогнозе". Требуется, чтобы
   карты были единым организмом с прогнозом и месячной картиной.
4. Данные в текущем экспортном JSON расходятся с документами банков.

Это **этап 1** из трёх. Этапы 2 (движок сценариев + вкладка "Сценарии" для ответа по
билетам) и 3 (переливы между картами, автоплатежи, стратегия перекрытия) фиксируются
как согласованный контур, но в этот спек не входят. Этап 1 закладывает под них
фундамент, не реализуя их.

## Целевая архитектура (контур всех этапов, для справки)

Вкладка "Сценарии" (вариант A): сценарий — это форк текущего состояния с наложенными
ходами (крупная покупка, заём с карты, новый кредит, разовый доход). Для форка
работает тот же `buildForecast`. Ядро: `applyScenario(state, moves) → state'` — чистая
функция. Это ложится на существующую трёхслойную архитектуру (чистое ядро → стор →
компоненты). Этап 1 не трогает сценарии, но модель карт должна быть достаточной, чтобы
этап 2 не потребовал повторной миграции.

## Модель карты (новые/изменённые поля)

Даты — ISO-строки `"YYYY-MM-DD"`, локальные (как всё в проекте, без сдвигов TZ).

```js
{
  // --- даты ближайшего/текущего цикла (вводятся явно, как в приложении банка) ---
  statementDate: "2026-07-26",   // дата выписки
  dueDate:       "2026-08-19",   // крайний срок обязательного платежа по этой выписке
  graceEndDate:  "2026-09-08",   // конец беспроцентного периода (может совпадать с dueDate)
  statementCycleDays: 30,        // длина расчётного периода, для автопродления вперёд

  // --- формула обязательного платежа ---
  minPaymentPercent: 4,                              // % базы
  minPaymentBase: "statement" | "currentDebt",       // от чего берётся %
  minPaymentFixed: { amount: 400, currency: "RUB" }, // "не менее"
  minPaymentPlusInterest: false,                     // добавлять ли проценты (Озон/Уралсиб — true)
  apr: 0.999,                                        // ставка годовых (доля, для процентов вне грейса)

  // --- суммы ---
  currentDebt:      { amount, currency },
  statementBalance: { amount, currency },
  payStrategy: "full" | "minimum",

  // --- задел под этап 2 (переливы); в этапе 1 только хранится, не используется в логике ---
  transferLimit:     { amount: 150000, currency: "RUB" }, // беспроцентный лимит переводов/снятия
  transferGraceDays: 55,                                  // грейс на перевод (дней)

  // существующие
  id, name, bank, owner, creditLimit, gracePeriodDays, disabled, note
}
```

Старые поля `statementDay`/`dueDay` удаляются из модели (мигрируются в даты). Поле
`gracePeriodDays` остаётся как справочное (инфо в UI), логика опирается на явные даты.

## Финансовое ядро (`src/finance.js`, чистые функции + тесты)

### cardCycle(card, from = today())

Возвращает актуальный на дату `from` цикл карты: `{ statement, due, graceEnd }` (объекты
Date). Логика:

- Стартует от сохранённых `statementDate/dueDate/graceEndDate`.
- Если `due < from` (цикл уже прошёл), прокручивает все три даты вперёд шагами по
  `statementCycleDays` (или помесячно, сохраняя якорный день выписки, если так проще и
  устойчивее к длине месяца), сохраняя смещения `due − statement` и `graceEnd − statement`
  постоянными относительно новой даты выписки.
- Возвращает первый цикл, у которого `due >= from`.

`cardNextDue(card, from)` переписывается как тонкая обёртка: `const { statement, due } =
cardCycle(card, from)`.

### cardMinPayment(card, rates)

```
база     = minPaymentBase === "statement" ? statementBalance : currentDebt   (в рублях)
byPct    = база × minPaymentPercent/100
fixed    = minPaymentFixed (в рублях)
core     = max(byPct, fixed)
interest = minPaymentPlusInterest ? apr × долг × statementCycleDays/365 : 0
result   = min(core + interest, долг)     // не больше долга
```

Проценты считаются простой формулой (согласовано): `apr × сумма × дни/365`.

Точные параметры из тарифов (проверено по документам):
- Т-Банк ТП 7.142 (муж) / ТП 7.143 (жена): минимальный платёж не более 14% от
  задолженности, минимум 600 ₽, кратно 100 ₽; ставка вне грейса 61,9% (`apr 0.619`).
  `plusInterest: false` (в грейсе покупки под 0%). Округление до сотен — опционально,
  не критично для решения; можно опустить в этапе 1.
- Озон: 4% + проценты, минимум 400 ₽; ставка вне грейса ~62,4% (`apr 0.624`).
- Уралсиб: 3% + проценты, минимум 300 ₽; ставка вне грейса 99,9% (`apr 0.999`).

Расхождение для этапа 2 (зафиксировано, на этап 1 не влияет): по тарифу Т-Банка
бесплатные переводы через сервисы банка — до 50 000 ₽ за расчётный период, сверх —
2,9% + 290 ₽. Пользователь сообщает про беспроцентный лимит 150 000 ₽ по подписке Pro.
Модель переводов и точный лимит уточняются на этапе 2; поля `transferLimit`/
`transferGraceDays` в этапе 1 только хранятся.

### buildForecast / buildMonthly

- `buildForecast`: блок карт берёт `due` и `graceEnd` из `cardCycle(card, start)` вместо
  реконструкции из чисел. Событие карты кладётся на `due`. Это устраняет хрупкость, из-за
  которой карты могли выпадать за горизонт.
- `buildMonthly`: включить минимальные платежи карт в `obligatory` (сейчас там только
  expenses + loans), чтобы профицит и цели считались честно с учётом карт.

## Стор и миграция (`src/store.js`)

`migrate()` для старых карт (есть `statementDay`/`dueDay`, нет `statementDate`):

- Синтезировать `statementDate/dueDate/graceEndDate` на ближайший цикл от сегодняшней
  даты, используя старые числа и `gracePeriodDays` как ориентир.
- Проставить дефолты новых полей (`minPaymentBase: "currentDebt"`,
  `minPaymentPlusInterest: false`, `apr` по банку или 0, `statementCycleDays: 30`,
  `transferLimit: 0`, `transferGraceDays: gracePeriodDays`).

Старые сохранения не должны ломаться.

## Тестирование (`test/finance.test.js`, node --test)

- `cardCycle()`: автопродление через границу месяца; кламп якорного дня к концу месяца;
  случай, когда сохранённый цикл уже в прошлом (прокрутка на N шагов).
- `cardMinPayment()`: три формулы (Т-Банк %-от-долга без процентов; Озон 4%+проценты,
  мин. 400; Уралсиб 3%+проценты, мин. 300); кламп "не больше долга".
- `migrate()`: старая карта (только statementDay/dueDay) → корректный синтез трёх дат и
  дефолтов.
- `buildForecast`: карта с ненулевым долгом реально попадает в события в пределах
  горизонта (регрессия на баг "кредитки не видны").

## Диагностика бага "кредитки не в прогнозе"

Перед фиксом — воспроизвести на реальных данных (systematic-debugging). Гипотеза: при
"числовой" модели `cardNextDue()` мог давать `due` за пределами горизонта прогноза (6
мес) или раньше `start`, из-за чего событие отсеивалось в `add()`. Переход на явные
`dueDate` из `cardCycle()` устраняет причину. Подтвердить воспроизведением до внесения
правки.

## Границы этапа 1 (YAGNI)

НЕ входит: движок сценариев `applyScenario`, вкладка "Сценарии", переливы между картами
и подсказки перекрытия, автоплатежи, амортизация кредита по графику. Всё это — этапы
2-3. Компоненты остаются тонкими: логика в ядре, `CardsView.vue` только редактирует
новые поля и отображает `cardCycle`/`cardMinPayment`.
