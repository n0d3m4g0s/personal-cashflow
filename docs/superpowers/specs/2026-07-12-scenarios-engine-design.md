# Этап 2: движок сценариев + вкладка "Сценарии" (ответ по билетам)

Дата: 2026-07-12
Статус: согласован, готов к плану реализации
Предшествует: docs/superpowers/specs/2026-07-12-credit-cards-real-dates-design.md (этап 1, в main)

## Контекст и цель

Этап 1 (в main) сделал карты единым организмом: реальные даты грейса, авторасчёт
минимального платежа, карты в прогнозе и месячной картине. Этап 2 строит поверх этого
**движок сценариев** и вкладку "Сценарии" (вариант A из брейнсторма этапа 1).

**Ключевая цель:** дать прямой ответ на реальный вопрос пользователя — как оплатить
билеты (~240-300k). Сравнить: перевод с карты жены (150k беспроцентно 55 дней, Pro
подключена), собственные наличные, новый кредит. Показать, укладывается ли возврат
займа в грейс и какой способ дешевле/безопаснее.

## Архитектура

Сценарий — именованный набор **ходов** (moves), наложенный на текущее состояние.
Новое чистое ядро `src/scenarios.js` (тестируется отдельно, `finance.js` не трогаем):

- `applyScenario(state, scenario) → state'` — копия state с применёнными ходами
  (разовые доходы/расходы, займы, кредиты транслируются в обычные записи
  incomes/expenses/loans/cards). НЕ мутирует исходный state.
- `evaluateScenario(state, scenario, opts) → { forecast, metrics }` — прогоняет
  `applyScenario` + существующий `buildForecast(state')` и вычисляет метрики для таблицы.

Принцип: ход → обычная запись в форке → тот же `buildForecast`. Движок прогноза этапа 1
не меняется, добавляется только слой поверх.

## Модель ходов (moves)

```js
// scenario = { id, name, moves: [...], baseFrom: '2026-07-18' | null }

// 1. Крупная покупка → разовый расход
{ type: 'purchase', title, amount: {amount, currency}, date }

// 2. Заём с карты → +наличные сейчас, рост долга карты; возврат авто или вручную
{ type: 'cardLoan', cardId, amount: {amount, currency}, date,
  repay: 'auto' | { date } }

// 3. Новый кредит → аннуитетные платежи, упрощённые проценты
{ type: 'newLoan', title, amount: {amount, currency}, apr, termMonths, startDate }

// 4. Разовый доход/корректировка → разовый income(+)/expense(−)
{ type: 'adjust', title, amount: {amount, currency}, sign: 1 | -1, date }
```

### Трансляция ходов в состояние (applyScenario)

- `purchase` → разовый expense `{ schedule: { frequency:'once', startDate: date } }`.
- `cardLoan` → (а) разовый income `+amount` на `date` (наличные на руках);
  (б) на карте `cardId`: `currentDebt += amount`; (в) событие возврата (см. ниже).
- `newLoan` → запись в loans с помесячным платежом (аннуитет), `remainingBalance = amount`,
  `paymentDay` из `startDate`, срок `termMonths`.
- `adjust` → разовый income (sign +1) или expense (sign -1) на `date`.

### Расчёт займа с карты (сердце сценария билетов)

Лимит беспроцентных переводов карты — поле `transferLimit` (этап 1; у карты жены 150000),
грейс на перевод — `transferGraceDays` (55) от даты займа; конец грейса =
`date + transferGraceDays`.

- В дату займа: `+amount` наличными, долг карты растёт на `amount`.
- **Часть в пределах лимита** `free = min(amount, transferLimit)`: беспроцентна, если
  возврат ≤ конца грейса перевода. Если возврат позже — проценты на `free` за дни сверх
  грейса: `apr_карты × free × (дни_сверх)/365`.
- **Часть сверх лимита** `over = max(0, amount − transferLimit)`: проценты С ПЕРВОГО ДНЯ
  займа по дату возврата: `apr_карты × over × (дни_займа_до_возврата)/365`.
- Итоговая переплата займа = проценты(free) + проценты(over).

**Возврат `auto`:** запас прочности определяется балансом прогноза. Идём по дням форка
(из `buildForecast`, где `cardLoan` уже дал +amount наличными, но событие возврата ещё не
поставлено); дата возврата — первый день ПОСЛЕ даты займа, на который `balance` того дня
≥ `startingCash + amount` (т.е. накопилось достаточно свыше стартового кэша, чтобы вернуть
заём, не проваливая исходную позицию). Ставим на эту дату событие `−amount` (гашение долга
карты) и пересчитываем. Если такой день не найден до конца горизонта — заём помечается
`repaid:false`, дата возврата = конец горизонта, проценты считаются до конца горизонта.
Определение через `balance` (а не через суммирование dayTotal) устойчивее: учитывает
стартовый кэш и все прочие движения.

**Возврат `{ date }`:** возврат ставится на указанную дату; проценты по формуле выше.

### Расчёт нового кредита

Аннуитетный платёж по стандартной формуле; для метрики переплаты — сумма процентов из
помесячного графика (остаток × apr/12 за каждый месяц). Точность до графика не требуется,
но помесячный проход точнее, чем грубая аппроксимация — используем помесячный.

## Метрики (evaluateScenario)

Для таблицы сравнения:

- `minBalance` — минимум balance за горизонт (красный, если < safetyBuffer).
- `minBalanceDate` — дата просадки.
- `overpayment` — суммарные проценты по всем cardLoan + newLoan сценария (рубли).
- `graceOk` — массив по каждому cardLoan: уложился ли возврат в грейс перевода (bool).
- `breakEvenDate` — первая дата, когда balance стабильно ≥ startingCash (или null).
- `risk` — 'низкий' | 'средний' | 'высокий':
  - высокий: minBalance < 0;
  - средний: 0 ≤ minBalance < safetyBuffer;
  - низкий: minBalance ≥ safetyBuffer.

## Вкладка "Сценарии"

- Новая вкладка в `App.vue` (значение `scenarios`), компонент `ScenariosView.vue`.
- `state.scenarios = []` (персистится как всё; `migrate` добивает пустой массив).
- Список сценариев: имя + редактор ходов (добавить/убрать ход; каждый тип — мини-форма
  с MoneyInput и датами).
- Таблица сравнения выбранных сценариев в колонках; строки — метрики выше. Первая
  колонка — базовый сценарий "Как есть" (без ходов) для опоры.
- Компоненты тонкие: вся логика в `scenarios.js`, читают/зовут ядро.

## Сид

`makeSeed()` добавляет один готовый сценарий "Билеты" с ходами: покупка 300k наличными
(date ~ через неделю) + заём 150k с карты жены (repay:'auto') + при необходимости adjust.
Чтобы пользователь сразу увидел ответ.

## Тестирование (`test/scenarios.test.js`, node --test)

- `applyScenario`: трансляция каждого типа хода; форк не мутирует исходный state.
- Заём с карты, возврат в грейс → overpayment 0.
- Заём с карты, возврат после грейса → проценты за дни сверх.
- Заём сверх transferLimit → проценты только на превышение (с первого дня).
- Авто-возврат: находит корректную дату из накопленного профицита; если не хватает —
  repaid:false, проценты до конца горизонта.
- `evaluateScenario`: метрики (minBalance, overpayment, breakEvenDate, risk) на известном
  входе.
- Сквозной тест сценария билетов на реалистичных числах: покупка 300k + заём 150k с карты
  жены при заданном профиците → возврат в грейс, overpayment 0 (или явная цифра).

## Границы этапа 2 (YAGNI)

НЕ входит: переливы между картами и подсказки перекрытия, автоплатежи, оптимизатор
порядка погашения (этап 3); полный график кредита до копейки; SVG-графики (только таблица
метрик). transferLimit учитывается для займа с карты; общая модель переводов между
картами — этап 3.
