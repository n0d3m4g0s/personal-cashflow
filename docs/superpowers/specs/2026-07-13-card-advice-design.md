# Этап 3c: оптимизатор-советник (рекомендации по картам)

Дата: 2026-07-13
Статус: согласован, готов к плану реализации
Предшествует: этапы 1, 2, 3a, 3b-1, 3b-2 — все в main. Финальный этап исходной задачи.

## Контекст и цель

3c — оптимизатор-советник: ранжированные объяснимые рекомендации по картам. Сначала
критические (риск просрочки/кассовый разрыв), потом экономия (перенос дорогого долга под
грейс). Каждая рекомендация несёт обоснование "почему" и опционально готовый ход (перелив)
с кнопкой "проиграть в Сценариях". Подход — ПРАВИЛА-ЭВРИСТИКИ (не перебор), для
прозрачности и предсказуемости. Блок "Рекомендации" во вкладке "Стратегия".

## Чистая функция `cardAdvice` (`src/advice.js`, новый файл)

`advice.js` — слой-советник поверх ядра. Может импортировать finance.js и scenarios.js
(направление advice→{finance,scenarios}, цикла нет). НЕ ядро.

```js
cardAdvice(state, opts = {}) → [{ severity, kind, title, why, action }]
```

- `severity`: 'critical' (риск просрочки) | 'warning' (внимание) | 'save' (экономия).
- `kind`: 'shortfall' | 'grace-ending' | 'transfer-save' | 'payoff-order'.
- `title`: короткий заголовок (напр. "Озон: грейс кончается 08.09, денег не хватает").
- `why`: объяснение с цифрами (напр. "К 08.09 остаток 12 000 ₽ < платежа 39 400 ₽. Перелив
  на карту жены даёт грейс до 02.11 бесплатно").
- `action`: `{ type: 'transfer', fromCardId, toCardId, amount, date, repay: 'auto',
  repayDate: '' }` или null (для информационных).
- Сортировка: critical → warning → save; внутри save — по убыванию экономии.

## Правила-эвристики

Проверяются по `buildForecast(state)` и данным карт. Каждое правило — чистая проверка,
генерирующая 0+ рекомендаций.

### Правило 1 (critical): кассовый разрыв к платежу карты — `shortfall`
Для каждого card-события прогноза: если `balance` дня события < 0 (или < буфера) → есть
риск не заплатить. Рекомендация: не хватает `shortfall = буфер − balance` (или `−balance`)
к дате D. `action`: перелив недостающей суммы на карту-приёмник с грейсом на перевод и
свободным лимитом (см. подбор приёмника ниже), если такая есть; иначе action=null (только
предупреждение "нужны деньги").

### Правило 2 (warning): грейс кончается без полного погашения — `grace-ending`
Full-карта, у которой `graceEndDate` в пределах горизонта, и в прогнозе на дату grace
остаток < долга карты (не хватает на полное погашение) → предупреждение. `action`: перелив
на карту с грейсом на перевод, если есть.

### Правило 3 (save): перенос дорогого долга под грейс — `transfer-save`
Для каждой карты-источника с долгом > 0 и apr > 0 (Уралсиб 99.9%, Озон 62%): подобрать
приёмник (карта с transferGraceEnabled и availableLimit > 0). Экономия = проценты, которые
набежали бы на исходной карте за грейс-период приёмника, МИНУС цена перелива
(`transferCost(toCard, ...).total`). Если экономия > 0 → рекомендация save с
`why` = "перенос сэкономит ≈ X ₽ (проценты Y − комиссия Z)". `action`: ход transfer.
Порог: любая положительная экономия (после вычета комиссии). Сортировка save по убыванию
экономии.

### Правило 4 (save, инфо): порядок погашения — `payoff-order`
Если есть свободные деньги (месячный профицit > 0, из buildMonthly): подсказать гасить
первой самую дорогую карту по ставке / с ближайшим грейсом. Информационная, `action`=null.
Одна рекомендация (не по карте), минимально.

### Подбор карты-приёмника (для action-переливов)
Кандидаты: активные карты с `transferGraceEnabled === true` и
`availableLimit = min(transferLimit, creditLimit − currentDebt) > 0`. Выбрать с наибольшим
availableLimit (у семьи — карта жены). Сумма перелива = min(долг источника, availableLimit
приёмника). Если приёмников нет — action=null.

## Механизм "Проиграть в Сценариях"

Нужен способ из компонента (1) добавить сценарий и (2) переключить активную вкладку. Сейчас
активная вкладка — локальный `ref` в App.vue, компоненты её не видят. Решение: добавить в
`src/store.js` экспортируемый reactive `nav = reactive({ active: 'dashboard' })` с функцией
`goTab(key)`. App.vue использует `nav.active` вместо локального ref (минимальная правка:
заменить `const active = ref('dashboard')` на импорт nav; кнопки вкладок пишут `nav.active`).
Кнопка "в Сценарии": `addScenario({ name, moves: [action] }); goTab('scenarios')`.

Альтернатива (проще, если правка App рискованна): кнопка только добавляет сценарий в
state.scenarios и показывает подсказку "сценарий создан, откройте вкладку Сценарии" — без
авто-переключения. План выберет минимально инвазивный вариант; предпочтителен nav-reactive.

Регрессия при nav-reactive: правка App.vue меняет управление активной вкладкой. ОБЯЗАТЕЛЬНО
проверить, что ВСЕ вкладки продолжают переключаться кликом (ручная проверка в браузере),
активная подсвечивается, дефолт 'dashboard' сохранён. Это трогает работающую навигацию —
подойти аккуратно, при сомнении выбрать альтернативу с подсказкой.

## Тесты (`test/advice.test.js`, новый файл)

- Правило shortfall: состояние с кассовым разрывом к дате card-события → critical с action
  переливом (если есть приёмник с грейсом).
- Правило grace-ending: full-карта, грейс в горизонте, остаток < долга → warning.
- Правило transfer-save: дорогая карта (Уралсиб apr 0.999) + карта жены (transferGraceEnabled,
  свободный лимит) → save, экономия > 0, action transfer на карту жены.
- Порог экономии: если комиссия > сэкономленных процентов → рекомендация НЕ создаётся.
- Сортировка: critical перед warning перед save; save по убыванию экономии.
- Здоровое/пустое состояние (нет карт с долгом, нет рисков) → пустой массив.
- Подбор приёмника: если нет карт с transferGraceEnabled → action=null у переливных правил.

## Границы 3c (YAGNI)

НЕ входит: перебор комбинаций переливов (только правила), авто-применение рекомендаций
(только предложение + кнопка), сложная оптимизация порядка погашения (правило 4 —
информационное, одна строка). Компонент не юнит-тестируется (Vue), проверяется в браузере.
