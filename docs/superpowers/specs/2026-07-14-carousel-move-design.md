# Ход «Карусель» переливов между картами

Дата: 2026-07-14
Статус: согласован, готов к плану реализации
Предшествует: этапы 1-3c (движок сценариев с ходом transfer) — все в main.

## Контекст и проблема

Ход `transfer` (перенос долга) моделирует ОДИН перенос: гасит долг fromCardId и ставит
событие-расход −amount (возврат ЖИВЫМИ деньгами) на дату возврата. Это неверно для
«карусели» — реальной практики перекладывать долг между двумя картами Т-Банка (муж↔жена)
каждые ~50 дней, обновляя беспроцентный грейс. Карусель НЕ тратит живые деньги (долг
перекладывается новым переводом) и держит сумму под 0% вместо ставки карты. Текущий движок
на карусели: (а) рисует ложную яму в кассе (событие-расход −amount, которого в реальности
нет), (б) считает ложные проценты. Пользователь применяет карусель на практике — движок
обязан считать её честно.

## Модель хода `carousel`

```js
// Крутить долг amount между картами cardAId и cardBId (обе с грейсом на перевод),
// перекладывая каждые (грейс − запас) дней от startDate до конца горизонта.
{ type: 'carousel', cardAId, cardBId, amount: {amount, currency}, startDate }
```

Обе карты должны иметь `transferGraceEnabled: true` (грейс на перевод — Т-Банк). Движок сам
генерирует цепочку переводов A→B, B→A, A→B… с шагом `stepDays = min(grace A, grace B) − 5`
(запас до истечения грейса), пока дата ≤ конца горизонта.

## Что считает движок (честно)

1. **Кэш не тратится.** carousel НЕ добавляет событий-расходов в прогноз (в отличие от
   transfer). Долг перекладывается переводом с карты на карту, живые деньги не задействованы.
   Ложной ямы в кассе нет. applyScenario для carousel НЕ меняет currentDebt карт и НЕ
   добавляет income/expense (весь эффект — в метриках через carouselPlan).
2. **Проценты.** Пока каждый шаг ≤ грейса перевода — проценты 0. Если stepDays > грейса
   (вырожденный ввод) — проценты за просрочку по apr карты-держателя на сумму за дни сверх.
   При корректном stepDays (грейс − 5) проценты всегда 0.
3. **Комиссия.** Часть суммы сверх transferLimit на каждом обороте — комиссия
   `transferFeePercent/100 × over + transferFeeFixed` (в рублях). В пределах лимита — 0.
   Суммируется по всем оборотам.
4. **Проверка лимитов.** Ограничение реализуемости - ОБЩИЙ кредитный лимит приёмника, НЕ
   беспроцентный лимит перевода. На первом обороте хотя бы одна карта должна иметь свободный
   кредитный лимит `creditLimit − currentDebt ≥ amtRub` для приёма долга. Сумма СВЕРХ
   беспроцентного лимита перевода (`transferLimit`) карусель НЕ делает нереализуемой - на
   превышение считается комиссия (см. п.3). ВАЖНО: свободный лимит приёмника считается с
   учётом того, что долг сейчас "висит" на карте-отправителе (после переноса он уходит с
   отправителя на приёмник); далее долг перекладывается, лимиты освобождаются/занимаются
   симметрично (см. carouselPlan).
5. **Экономия.** `saved = amtRub × apr_holder × дни_в_горизонте / 365` — сколько сэкономлено
   за счёт 0% вместо ставки карты, на которой долг иначе висел бы. apr_holder — ставка карты,
   с которой стартует карусель (обычно дорогая карта-источник долга).
6. **Конец горизонта.** Долг остаётся на карте, где после последнего оборота (endHolder).
   Метрика показывает остаток долга под 0% — карусель это отсрочка, НЕ погашение.

## Чистая функция `carouselPlan`

```js
carouselPlan(cardA, cardB, amount, startDate, end, rates)
  → { transfers: [{ date, fromId, toId, graceEnd }], interest, fee, saved, feasible, warning, endHolderId }
```

- `stepDays = min(cardA.transferGraceDays, cardB.transferGraceDays) − 5` (минимум 1).
- Генерирует переводы: старт A→B на startDate, далее чередует направление каждые stepDays.
- Проверка лимитов (упрощённо, т.к. долг на каждом обороте симметрично уходит с одной карты
  и приходит на другую): ограничение реализуемости - ОБЩИЙ кредитный лимит приёмника, НЕ
  беспроцентный лимит перевода. Для муж↔жена: у жены свободно ~150к, у мужа свободно ~6к
  ИЗНАЧАЛЬНО - но карусель стартует с переноса ДОЛГА мужа на жену, после чего у мужа
  освобождается лимит. Далее лимиты чередуются автоматически (симметрия). Итог: `feasible`
  true, если (а) обе карты `transferGraceEnabled !== false`, (б) хотя бы одна карта имеет
  свободный кредитный лимит `creditLimit − currentDebt ≥ amtRub` для приёма на первом обороте.
  Сумма СВЕРХ `transferLimit` карусель НЕ ломает - даёт комиссию, не фейл.
- `interest`: сумма процентов по оборотам, где шаг вышел за грейс (0 при корректном шаге).
- `fee`: сумма комиссий по оборотам (`over × %/100 + fixed`, где `over = max(0, amtRub −
  transferLimit_приёмника)`; при сумме в пределах лимита перевода - 0). Суммируется по всем
  оборотам, идёт в overpayment.
- `saved`: amtRub × apr_holder × дни(startDate→end)/365, apr_holder = max(cardA.apr, cardB.apr)
  (долг иначе висел бы под старшей ставкой; для муж↔жена обе 61.9%).
- `feasible`/`warning`: false + текст, если у карты нет `transferGraceEnabled`, ИЛИ ни на одной
  карте нет свободного кредитного лимита ≥ amtRub для первого переноса. Сумма > `transferLimit`
  сама по себе НЕ делает feasible false.
- `endHolderId`: id карты, держащей долг после последнего оборота.

## Интеграция в `evaluateScenario`

- Ход carousel НЕ добавляет событий в forked (кэш не трогается) — в отличие от transfer.
- carouselPlan вызывается для расчёта метрик. `saved` идёт в НОВОЕ поле метрик
  `carouselSaved` (сумма по всем каруселям). `fee + interest` добавляется в `overpayment`.
- warning (нереализуемая карусель) собирается в `transferWarnings` (переиспользуем массив
  предупреждений) или новое `carouselWarnings`.

## Редактор хода в `ScenariosView.vue`

- Кнопка «+ Карусель» рядом с «+ Перенос долга».
- Blank-объект: `{ type: 'carousel', cardAId: state.cards[0]?.id, cardBId: state.cards[0]?.id,
  amount: {amount:0,currency:'RUB'}, startDate: '' }`.
- Форма: два селекта карт (A и B из state.cards), MoneyInput (сумма), input date (старт).
- Название типа: 'Карусель переливов'.
- В таблице сравнения: строка «Экономия карусели» (money(carouselSaved)).

  interest 0, fee 0, saved > 0, feasible true. Проверить чередование направлений и шаг ~50д.

- carouselPlan: карта без transferGraceEnabled → feasible false, warning.

  от карусели (нет ложной ямы, в отличие от transfer). Сравнить: сценарий с transfer даёт
  просадку кассы, тот же с carousel — нет.
- Регрессия: существующие тесты transfer/cardLoan/scenarios зелёные.

## Границы (YAGNI)

Карусель только между ДВУМЯ картами (не 3+). Шаг фиксированный (грейс − 5 дней), без
настройки запаса. Долг в конце висит; гашение — отдельный ход adjust при желании. Комиссия
и проценты считаются, но для карт пользователя (в лимите 150к, обе Т-Банк) обычно 0.
